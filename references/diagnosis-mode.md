# 诊断模式详细流程

## 模式概述

诊断模式（Diagnose Mode）采用**自下而上**的分析路径：从用户感知到的问题出发，逐层追溯到根本原因。

```
用户感知的问题（现象）
    ↓
体验层检查 → 用户旅程哪个环节出问题了？
    ↓
转化层检查 → 流程/能力出了什么问题？
    ↓
逻辑层检查 → 定位/价值/模式有什么问题？
    ↓
根因分析 → 最核心的原因是什么？
    ↓
修复方案 → 怎么修？
```

## Step 1: 问题定位

### 信息收集

在开始诊断之前，需要收集足够的信息。

**必问问题**：

| 问题 | 目的 | 示例 |
|------|------|------|
| 发生了什么？ | 了解现象 | "用户增长从每月15%降到2%" |
| 什么时候开始的？ | 时间线定位 | "最近3个月" |
| 影响了谁？ | 用户群体定位 | "主要是新用户" |
| 严重程度？ | 影响范围 | "月活从10万降到8万" |
| 做过什么尝试？ | 已有措施 | "加了推荐功能但没效果" |
| 有没有数据支撑？ | 量化依据 | "注册转化率从30%降到15%" |

### 初步分类

根据用户的描述，初步判断问题偏向哪一层：

| 用户说的话 | 偏向层面 | 判断依据 |
|-----------|---------|---------|
| "不好用"、"太复杂"、"不美观" | 体验层 | 用户直接感知到的 |
| "做不出来"、"效率低"、"质量差" | 转化层 | 产出能力相关 |
| "没人要"、"没市场"、"方向错了" | 逻辑层 | 商业本质相关 |
| "增长停滞"、"用户流失" | 可能跨层 | 需要逐层排查 |

**注意**: 初步分类只是方向，不是结论。真正的根因可能在任何一层。

## Step 2: 逐层诊断

### 2a. 体验层检查（首先检查）

**原因**: 体验层是最直接的，用户感知的问题通常在这里。

**检查清单**:

```
□ 用户旅程是否完整？
  - 接触 → 了解 → 试用 → 购买 → 使用 → 复购
  - 哪个环节断了？哪个环节体验差？

□ 关键触点是否达标？
  - 首次使用体验如何？（新手引导）
  - 核心操作是否流畅？（日常使用）
  - 遇到问题时有帮助吗？（客服/文档）

□ 体验底线是否守住？
  - 核心功能是否稳定？
  - 响应时间是否达标？
  - 数据是否安全？

□ 差异化体验是否存在？
  - 用户有没有"Wow"时刻？
  - 有没有让用户记住的点？
  - 竞品做得比我们好吗？
```

**诊断方法**:
- 分析用户评价/反馈
- 查看用户行为数据（漏斗分析）
- 对比竞品体验
- 检查 NPS/满意度数据

**常见发现**:

| 现象 | 可能原因 | 深层问题 |
|------|---------|---------|
| 新用户注册率低 | 注册流程复杂？ | 转化层/体验层 |
| 用户用一次就走 | 核心价值未感知？ | 逻辑层/体验层 |
| 用户反馈"难用" | 交互设计差？ | 体验层 |
| 用户说"没什么用" | 价值不够？ | 逻辑层 |

### 2b. 转化层检查

**如果体验层没有明显问题，或者问题指向更深层**，检查转化层。

**检查清单**:

```
□ 生产线是否完整？
  - 输入 → 转化 → 输出 → 反馈
  - 哪个环节有瓶颈？

□ 输入端
  - 需求输入是否充分？
  - 输入质量是否有保证？

□ 转化端
  - 核心流程是否高效？
  - 质量控制是否到位？
  - 人工依赖程度是否过高？

□ 输出端
  - 产出质量是否稳定？
  - 交付是否及时？
  - 产出形式是否符合用户期望？

□ 反馈端
  - 用户反馈是否有效收集？
  - 反馈是否驱动了改进？
  - 数据回流是否顺畅？
```

**诊断方法**:
- 流程效率分析
- 质量数据分析
- 成本分析
- 瓶颈识别

**常见发现**:

| 现象 | 可能原因 | 深层问题 |
|------|---------|---------|
| 产出质量不稳定 | 流程不标准？ | 转化层 |
| 成本太高 | 流程低效？人力过多？ | 转化层 |
| 交付不及时 | 瓶颈环节？ | 转化层 |
| 用户需求与产出不匹配 | 输入端理解偏差？ | 逻辑层/转化层 |

### 2c. 逻辑层检查（最后检查）

**如果体验层和转化层都没有根本性问题**，或者问题指向商业本质，检查逻辑层。

**检查清单**:

```
□ 定位是否正确？
  - 品类定义还成立吗？
  - 差异化还有效吗？

□ 价值主张是否成立？
  - 解决的问题还是真问题吗？
  - 用户痛点还在吗？
  - 有没有更好的替代方案出现了？

□ 目标人群是否准确？
  - 用户画像还准吗？
  - 市场规模有变化吗？
  - 付费意愿如何？

□ 商业模式是否可行？
  - 收入能覆盖成本吗？
  - 增长飞轮在转吗？
  - 竞争格局有变吗？

□ 竞争优势是否还在？
  - 壁垒有没有被突破？
  - 竞品有没有赶上来？
  - 新入局者有威胁吗？
```

**诊断方法**:
- 市场调研
- 竞品分析
- 商业模型验证
- 用户访谈

**常见发现**:

| 现象 | 可能原因 | 判断 |
|------|---------|------|
| 用户付费意愿低 | 价值主张不成立？定价错？ | 🔴 可能要改方向 |
| 市场在萎缩 | 赛道选错了？ | 🔴 可能要转型 |
| 竞品在追赶 | 壁垒不够深？ | 🟡 要加强壁垒 |
| 增长飞轮停了 | 某个环节失效？ | 🟡 要修复飞轮 |

## Step 3: 根因分析

### 5 Why 分析法

对最突出的问题，连续追问5个"为什么"。

**示例**:

```
问题: 月活从10万降到8万

Why 1: 为什么月活下降了？
→ 因为新用户注册后大部分不回来了

Why 2: 为什么新用户不回来？
→ 因为首次使用体验不好，核心功能找不到

Why 3: 为什么首次体验不好？
→ 因为新手引导设计有问题，信息过载

Why 4: 为什么新手引导设计有问题？
→ 因为设计时没做过用户测试

Why 5: 为什么没做用户测试？
→ 因为团队急于上线，跳过了用户研究环节
→ 根因: 团队节奏过快，忽略了用户体验验证
```

### 根因分类

| 根因类型 | 特征 | 修复方向 |
|---------|------|---------|
| 逻辑层根因 | 定位/价值/模式有问题 | 可能需要转型或调整方向 |
| 转化层根因 | 流程/能力/效率有问题 | 优化流程，提升能力 |
| 体验层根因 | 设计/交互/触点有问题 | 重新设计用户体验 |
| 跨层根因 | 多个层面都有问题 | 需要系统性修复 |

## Step 4: 修复方案

### 优先级分类

```
🔴 P0 - 紧急修复（不修会死）
  → 影响生存的问题
  → 必须立即处理

🟡 P1 - 重要优化（影响增长）
  → 影响竞争力的问题
  → 1-2周内处理

🟢 P2 - 锦上添花（提升体验）
  → 优化体验的问题
  → 有时间再处理
```

### 修复方案模板

每个修复方案必须包含以下全部要素：

```
问题: [具体问题描述]
根因: [5 Why 找到的根本原因]
方案: [具体做什么]
预期效果: [做了之后会怎样 — 必须有推导过程，不能直接写数字]
实施难度: 高/中/低
时间周期: [需要多久]
成本估算: [人力/资金/资源]
投入产出比: [如果无法估算，说明需要什么信息]
验证指标: [可量化指标 + 验收标准 + 验收时间]
```

### 多问题场景的 ROI 处理

当多个修复方案相互依赖时，必须进行捆绑评估：

1. **捆绑 ROI 推导**: 将相互依赖的方案捆绑为一个"修复组合"，用场景数据计算实际金额（不能使用占位符）
2. **依赖链标注**: 标注方案之间的依赖关系（如"P1的ROI取决于P0的执行效果"）
3. **最小可验证组合**: 识别"最小一组修复方案"——只需执行这组方案就能验证核心假设是否成立
4. **无法独立估算时**: 必须写明"此方案的ROI取决于[具体前置方案]，需先完成后才能估算 ❓"

### Step 4.5: 一致性检查（诊断模式）

在输出修复方案后，检查诊断结论和修复方案之间是否一致：
- 根因→修复对齐：修复方案是否真的解决了5 Why分析出的根因？
- 修复方案间矛盾：不同优先级的修复方案之间是否有冲突？
- 数据→结论对齐：诊断结论是否有数据支撑？还是基于推测？

## 常见诊断场景

### 场景 1: 用户增长停滞

```
典型根因:
- 体验层: 获客渠道体验差、推荐机制失效
- 转化层: 获客效率低、转化流程有瓶颈
- 逻辑层: 市场饱和、定位不清晰、竞争力下降

排查顺序:
1. 检查漏斗数据，定位哪个环节流失最大
2. 分析竞品变化，看是否被抢占了份额
3. 检查获客渠道效率，看是否需要拓展新渠道
```

### 场景 2: 用户流失率上升

```
典型根因:
- 体验层: 核心功能体验退化、新版本不适应
- 转化层: 服务质量下降、响应变慢
- 逻辑层: 价值主张被竞品替代、用户需求变化

排查顺序:
1. 对比流失用户和留存用户的行为差异
2. 检查最近的产品变更
3. 收集流失用户的直接反馈
```

### 场景 3: 转化率下降

```
典型根因:
- 体验层: 注册/支付流程变复杂、信任感不足
- 转化层: 定价策略不当、交付承诺不清晰
- 逻辑层: 价值主张不够有吸引力、目标人群偏移

排查顺序:
1. 检查转化漏斗，定位流失最大的环节
2. A/B 测试不同方案
3. 分析用户决策路径
```

### 场景 4: 付费意愿低

```
典型根因:
- 体验层: 用户未感受到价值就要求付费
- 转化层: 定价过高/过低、付费流程不顺畅
- 逻辑层: 核心价值不成立、免费替代方案太多

排查顺序:
1. 分析免费→付费转化路径
2. 了解用户对价值的认知
3. 对比竞品定价
```

## 注意事项

1. **不要预设结论**: 虽然有初步分类，但要保持开放心态
2. **用数据说话**: 每个判断都要有数据支撑
3. **区分表象和根因**: 表面问题可能不是根本原因
4. **系统性思考**: 一个问题可能涉及多层
5. **给出可执行方案**: 修复方案要具体到可以执行
